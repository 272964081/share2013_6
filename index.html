"简单"背后

  • 未登录首页：2个版本
  • 登录首页: 最多时6个版本
  • 小组首页：最多时5个版本
  • 广播类型全集：195种
  • 提醒类型全集：67种
  • 小组登录提示：相继5个版本测试
  • 几乎所有新功能都要灰度上线
  • 移动化
  • 特点：新旧代码并存、业务逻辑复杂、迭代快、需求多变、向前兼容、移动设备友好
  • 不希望用人海战术解决这些问题

工程问题的产生

  • 新旧代码并存时，代码冗余问题、静态文件管理
  • 业务逻辑复杂时，通用和业务代码的分离、复杂度控制（代码架构问题）
  • 需求多变时，大而全的通用组件无用武之地，轻量的、功能单一的更便于复用
  • 迭代越快，对代码复用、模块化组织要求越高
  • 追求高效同时，如何保证代码质量
  • 追求高效同时，如何保证前后端性能最优
  • 追求高效同时，不能忽视安全问题
  • 追求高效同时，如何保证人的发挥空间
  • 通常利用工具提高生产力，怎样设计有效的工具链:
    1. "工具防止人做愚蠢的事，也阻碍人干聪明的事"
    2. 工具不是越强大越好，而是简单有效最好
    3. 工具不是越傻瓜越好，要留给他人发挥的空间
    4. 提高生产力同时激发创造力
  • 响应式开发。网页的适配性、灵活的输出（重新思考）
  • 团队协作模式

总结: 工程"支柱"问题

工程开发经验：解决工程问题的经验

例子: 在全站导航上加一个消息提醒

技术问题：提示框UI和交互的实现

工程问题：

  1. 导航是跨产品线
  2. 各产品可能是在不同的代码仓库中
  3. 它是临时的一周后撤下
  4. 只有首次注册的新用户才能看到
  5. 它有多个版本测试使用效果
  6. 它不能影响页面的性能

前端开发80％是工程问题，20％是技术问题

  • Javascript:

    IE6(JScript5.6, ES3)/IE7(JScript5.7, ES3)/IE8(JScript5.8, ES3)/IE9,10(JScript.NET, ES5/ES5.1)
    Firefox 4+(ES5/5.1)/Chrome1~10(ES3), Chrome10+(ES5/5.1)/Safari3~5(ES3), Safari5+(ES5)/Opera11+(ES5)
  • CSS:
    CSS2.1，CSS的稳定版
    CSS3是以CSS2.1为核心，发展或替换其中的模块。关注各个模块的进展和浏览器支持情况
  • HTML:HTML 4.01、HTML 5.1
  • SVG1.1
  • Javascript APIs
  • Web Components
  • DOM(Level 1, 2, 3)
  • 更多: http://www.w3.org/TR/

一些经验...

  • 模块要完全独立(借助工具实现)
  • 通用代码中绝不混杂业务逻辑
  • 业务逻辑复杂时，应该按业务类型拆分，不是按展现形式拆分
  • 代码架构借鉴SOLID原则:

    S - Single Responsibility Principle 职责分离原则
    O - Open/Closed Principle 开闭原则
    L - Liskov Substitution Principle 里氏替换原则
    I - Interface Segregation Principle 接口分离
    D - Dependency Inversion Principle 依赖反转原则

  • 从实际开发中积累形成的生态体系
  • 技术问题上开放，工程问题上保守
  • 在完成的基础上追求完美
  • 工具要简单、配置要简单
  • 工具是可以替换的，而且总是在变的，不要成为工具的奴隶
    "我们的发明常常是漂亮的工具，只是吸引我们的注意力，使我们离开了严肃的事物。" -- 梭罗《瓦尔登湖》
  • 对于复杂的问题，不断进行拆解直到足够简单
  • 原研哉的Exformation哲学，受用！
  • 更多的时间做有趣的事!
