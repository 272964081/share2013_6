"简单"背后

  • 未登录首页：2个版本
  • 登录首页: 最多时6个版本
  • 小组首页：最多时5个版本
  • 广播类型全集：195种
  • 提醒类型全集：67种
  • 小组登录提示：相继5个版本测试
  • 几乎所有新功能都要灰度上线
  • 移动化的策略
  • 特点：新旧代码并存、业务逻辑复杂、迭代快、需求多变、向前兼容、移动设备友好
  • 不希望用人海战术解决这些问题

自然产生的工程问题

  • 新旧代码并存 >>>>>>>> 代码冗余问题、静态文件管理
  • 业务逻辑复杂 >>>>>>>> 通用和业务代码的分离、复杂度控制（代码架构问题）
  • 需求多变 >>>>>>>>>>> 大而全的通用组件无用武之地，轻量的/单一功能的更便于复用

    当有人问我通用组件的版本管理时，我才意识到这对于我来说早已不是问题。粒度越小越稳定，不能满足需求时通常也不去改它，而是扩展或替换。

  • 迭代周期短/频率高 >>>> 对代码复用、模块化组织要求越高
  • 追求高效 <<<<<<<<<<< 如何保证代码质量
  • 追求高效 <<<<<<<<<<< 如何保证前后端性能最优
  • 追求高效 <<<<<<<<<<< 不能忽视安全问题
  • 追求高效 <<<<<<<<<<< 如何保证人的发挥空间
  • 通常利用工具提高生产力，怎样设计有效的工具链(工具的负面作用):

    1. "工具防止人做愚蠢的事，也阻碍人干聪明的事"
    2. 工具不是越强大越好，而是简单有效最好
    3. 工具不是越傻瓜越好，要留给他人发挥的空间
    4. 提高生产力同时激发创造力

  • 团队内部的技术驱动力 >>>> 响应式开发。网页的适配性、灵活的输出（重新思考网页）
  • 团队协作模式 >>>>>>>>>>> 适合团队特点的工作流

技术问题是什么

  • 技术问题是标准问题 (纯粹前端技术===W3C/ECMA-262制订的标准, 民间/企业标准除外, 如commonjs, open graph...)
  • 学习前端技术从阅读标准开始
  • Javascript:
    1998.10, ECMA-262 1st Edition(ES1), Javascript标准化的开始
    IE6(JScript5.6, ES3)/IE7(JScript5.7, ES3)/IE8(JScript5.8, ES3)/IE9,10(JScript.NET, ES5/ES5.1)
    Firefox 4+(ES5/5.1)/Chrome1~10(ES3), Chrome10+(ES5/5.1)/Safari3~5(ES3), Safari5+(ES5)/Opera11+(ES5)
  • CSS:
    CSS2.1，CSS的稳定版
    CSS3是以CSS2.1为核心，发展或替换其中的模块。参照CSS2.1标准框架学习，关注各个模块的进展和浏览器支持情况
  • HTML:HTML 4.01、HTML 5.1
  • SVG1.1
  • Javascript APIs
  • Web Components
  • DOM(Level 1, 2, 3)
  • 更多: http://www.w3.org/TR/

一些经验...

  • 模块要完全独立(借助工具实现)
  • 通用代码中绝不混杂业务逻辑
  • 业务逻辑复杂时，应该按业务类型拆分，不是按展现形式拆分
  • 代码架构领会SOLID原则:

    S - Single Responsibility Principle 职责分离原则
    O - Open/Closed Principle 开闭原则
    L - Liskov Substitution Principle 里氏替换原则
    I - Interface Segregation Principle 接口分离
    D - Dependency Inversion Principle 依赖反转原则

  • 代码架构设计的目的性

    "A good architecture allows volatile decisions to be easily changed" - Bob大叔

  • 从实际开发中积累形成的生态体系
  • 技术问题上开放，工程问题上保守
  • 在完成的基础上追求完美
  • 工具要简单、配置要简单
  • 工具是可以替换的，而且总是在变的，不要成为工具的奴隶

    "我们的发明常常是漂亮的工具，只是吸引我们的注意力，使我们离开了严肃的事物。" -- 梭罗《瓦尔登湖》

    "是否符合潮流，是否与其它形式相关或孤立都不重要。重要的是，这个形式的产生，是否基于内在的需要！" -- 康定斯基《关于形式问题》

    形式越重，越容易忽视本质

  • 对于复杂的问题，不断进行拆解直到足够简单
    复杂问题是简单问题的积分
  • 原研哉的Exformation哲学，受用！
  • 更多的时间做有趣的事!
